Etimología

Algoritmo viene del nombre latinizado del matemático persa al-Khwarizmi. Mucho antes de las computadoras electrónicas, un algoritmo era un procedimiento confiable para calcular. Programación es escribir un plan ejecutable por adelantado.

Programación y algoritmos convierten intención en procedimiento exacto: una idea vaga debe volverse suficientemente clara para que una máquina la ejecute y otra persona la verifique.

Computabilidad

La base científica de la computación cambió con Alan Turing y Alonzo Church. La máquina de Turing no era una pieza de hardware. Era un modelo abstracto de procedimiento: símbolos, estados, reglas y una cinta. Eso permitió preguntar qué problemas pueden computarse.

La computabilidad marca la primera frontera. Algunas tareas no son resolubles por algoritmo en general. El problema de la parada es la advertencia clásica: no existe un algoritmo universal que decida, para todo programa y entrada, si ese programa se detendrá.

Corrección antes de velocidad

Un programa puede parecer correcto porque el caso de prueba era fácil. Un algoritmo debe funcionar porque su estructura fuerza el resultado correcto. La corrección, los invariantes, las precondiciones y las postcondiciones hacen explícita esa estructura.

Esto dice qué debe ser verdadero antes de ejecutar código y qué debe ser verdadero al terminar. Es la versión matemática de no aceptar un programa solo porque pareció convincente una vez.

Complejidad y Big O

La corrección pregunta si el algoritmo da la respuesta correcta. La complejidad pregunta si puede hacerlo con recursos razonables. Tiempo, memoria, comunicación y energía forman parte del diseño.

Big O no es un cronómetro. Es un modelo de crecimiento. Explica por qué un método ineficiente puede parecer aceptable con entradas pequeñas y colapsar cuando la entrada crece.

Programación dinámica y decisiones voraces

La programación dinámica guarda respuestas a subproblemas para no recalcularlas. Funciona cuando un problema tiene subproblemas repetidos y subestructura óptima.

Los algoritmos voraces son distintos: toman la mejor decisión local y dependen de una estructura que demuestre que esa decisión local es globalmente segura. Dijkstra es un algoritmo voraz. No todo problema admite una estrategia voraz.

P, NP (tiempo polinomial no determinista) y dificultad

La teoría de complejidad estudia los límites de la computación eficiente. Algunos problemas son resolubles eficientemente. Otros tienen soluciones verificables eficientemente, pero no conocemos una forma eficiente de encontrarlas.

Los problemas NP-completos son centrales porque una solución en tiempo polinomial para cualquiera de ellos daría soluciones en tiempo polinomial para todos los problemas en NP. Las reducciones muestran cuándo un problema carga la dificultad de otro.

Programar como disciplina de ingeniería

La programación real añade restricciones que el pseudocódigo puede ocultar: casos extremos, entradas inválidas, distribución de memoria, concurrencia, latencia, límites de API (interfaz de programacion de aplicaciones) (application programming interfaces, interfaces de programación de aplicaciones), corrupción de datos y mantenibilidad. Buen software no es solo inteligente. Es legible, comprobable y honesto sobre sus fallas.

Los algoritmos dan estructura; la programación hace que esa estructura sobreviva al contacto con sistemas reales.

Glosario de Términos Técnicos

  • ▣Invariante: Una afirmación que permanece verdadera durante cada paso de un algoritmo y ayuda a probar corrección.
  • ▣Precondición: Lo que debe ser cierto antes de ejecutar un algoritmo o función para que su garantía aplique.
  • ▣Postcondición: Lo que debe ser cierto después de que un algoritmo o función termina si es correcto.
  • ▣Recurrencia: Una ecuación que describe el tiempo de ejecución relacionando un problema con subproblemas más pequeños.
  • ▣Cota inferior: Una prueba de que ningún algoritmo dentro de un modelo puede superar cierto costo mínimo.
  • ▣Reducción: Una transformación que muestra que resolver un problema también resolvería otro.
  • ▣Frontera: El borde descubierto pero aún no procesado completamente dentro de una búsqueda en grafo.
  • ▣Tratable: Suficientemente práctico para resolverse con recursos razonables, a menudo asociado con algoritmos de tiempo polinomial.
  • ▣Intratable: Demasiado costoso para resolverse exactamente a escala útil con los métodos conocidos.
