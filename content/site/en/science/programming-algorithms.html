Etymology

Algorithm comes from the Latinized name of the Persian mathematician al-Khwarizmi. Long before electronic computers, an algorithm meant a reliable procedure for calculation. Programming is the act of writing an executable plan in advance.

Programming and algorithms turn intent into exact procedure: a vague idea must become clear enough for a machine to execute and for another person to verify.

Source of etymological origin: Online Etymology Dictionary: algorithm; Online Etymology Dictionary: program.

Computability

The scientific foundation of computation changed with Alan Turing and Alonzo Church. Turing's machine was not a piece of hardware. It was an abstract model of procedure: symbols, states, rules, and a tape. That made it possible to ask which problems can be computed at all.

Computability sets the first boundary. Some tasks are not algorithmically solvable in general. The halting problem is the classic warning: there is no universal algorithm that can decide for every program and input whether that program eventually stops.

Correctness Before Speed

A program can appear to work because the test case was easy. An algorithm should work because its structure forces the right result. Correctness, invariants, preconditions, and postconditions make that structure explicit.

This says what must be true before code runs and what must be true after it finishes. It is the mathematical version of not accepting a program just because it looked convincing once.

Complexity and Big O

Correctness asks whether an algorithm gives the right answer. Complexity asks whether it can do so with reasonable resources. Time, memory, communication, and energy all become part of the design.

Big O is not a stopwatch. It is a growth model. It explains why an inefficient method can feel acceptable on small inputs and collapse when the input becomes large.

Dynamic Programming and Greedy Choices

Dynamic programming stores answers to subproblems so they are not recomputed. It works when a problem has overlapping subproblems and optimal substructure.

Greedy algorithms are different: they make the best local choice and rely on structure proving that local choice is globally safe. Dijkstra is greedy. Not every problem allows greed.

P, NP (nondeterministic polynomial time) and Hardness

Complexity theory studies the limits of efficient computation. Some problems are efficiently solvable. Some have solutions that are efficiently checkable but no known efficient way to find them.

NP-complete problems are central because a polynomial-time solution to any one of them would give polynomial-time solutions to all problems in NP. Reductions show when one problem carries the difficulty of another.

Programming as Engineering Discipline

Real programming adds constraints that pure pseudocode can hide: edge cases, invalid input, memory layout, concurrency, latency, API (application programming interface) boundaries, data corruption, and maintainability. Good software is not only clever. It is readable, testable, and honest about failure.

Algorithms give structure; programming makes that structure survive contact with real systems.

Modern Frontier

Modern programming is not just writing commands. It includes algorithms, data structures, formal verification, distributed systems, security, numerical reliability, complexity limits, compiler design, and reproducible engineering. The frontier includes verified software, proof assistants, safe systems languages, privacy-preserving computation, and algorithms that must remain correct under scale and failure.

AI-assisted coding changes workflow but not the standard. Code still has to be tested, reviewed, secured, profiled, documented, and understood. A program that appears to work on one example is not the same as a correct algorithm.

Technical Terms Glossary

  • ▣Invariant: A statement that stays true through each step of an algorithm and helps prove correctness.
  • ▣Precondition: What must be true before an algorithm or function runs for its guarantee to apply.
  • ▣Postcondition: What must be true after an algorithm or function finishes if it is correct.
  • ▣Recurrence: An equation that describes runtime by relating a problem to smaller subproblems.
  • ▣Lower bound: A proof that no algorithm in a model can beat a certain minimum cost.
  • ▣Reduction: A transformation showing that solving one problem would also solve another.
  • ▣Frontier: The currently discovered but not fully processed boundary of a graph search.
  • ▣Tractable: Practical enough to solve within a reasonable resource bound, often associated with polynomial-time algorithms.
  • ▣Intractable: Too expensive to solve exactly at useful scale with known methods.
